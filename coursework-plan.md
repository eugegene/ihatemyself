## РОЗДІЛ 1. АНАЛІЗ ОБ'ЄКТУ ТЕСТУВАННЯ

### 1.1. Загальний опис системи
Застосунок Track List — платформа для відстеження та рецензування медіаконтенту (фільми, серіали, книги). Курсова робота розглядає виключно frontend частину: клієнтський та серверний код SvelteKit, утиліти, компоненти та store.

### 1.2. Архітектура frontend

- **Фреймворк:** SvelteKit 2 (SSR + CSR)
- **Мова:** TypeScript (strict mode)
- **Стилізація:** Tailwind CSS 4
- **Маршрутизація:** файлова система (`src/routes/`)
- **Стан застосунку:** Svelte 5 runes store (`src/lib/stores/`)
- **SSR middleware:** `hooks.server.ts` — перевірка JWT на кожен запит

### 1.3. Модулі, що підлягають тестуванню

| Категорія | Модулі | Файли |
|---|---|---|
| Утиліти | API клієнт, хелпери, валідація, JWT-claims, localStorage | `src/lib/utils/`, `src/lib/api.ts` |
| Серверна логіка | Авторизація, cookie, token refresh | `src/lib/server/auth.ts`, `src/hooks.server.ts` |
| UI компоненти | Header, Footer | `src/lib/components/` |
| Store | userStore (Svelte 5 runes) | `src/lib/stores/user.svelte.ts` |
| Сторінки | Сторінка помилок | `src/routes/+error.svelte` |

### 1.4. Технологічний стек тестування

| Рівень | Інструмент | Призначення |
|---|---|---|
| Модульне | Vitest 4.x + jsdom | Ізольоване тестування функцій та компонентів |
| Інтеграційне | Vitest + @testing-library/svelte | Тестування взаємодії між модулями |

### 3.3. Інтеграційні тест-кейси

#### I1: Token Refresh Flow (`hooks.server.ts` ↔ `auth.ts`)

**Передумова:**
- Мок: access token прострочений, refresh token валідний
- Мок: API `/auth/refresh` повертає новий access token

**Кроки виконання:**
1. Виклик `handle()` з прострочений access token у cookie
2. `hooks.server.ts` детектує помилку верифікації
3. Викликає `auth.ts::refreshToken()`
4. Отримує новий токен
5. Встановлює нові cookie
6. Продовжує обробку запиту

**Очікуваний результат:** event.locals містить нові дані користувача; cookie оновлено

#### I3: Header ↔ userStore

**Сценарій A — авторизований:**
- Передумова: `userStore.set({ username: "testuser", email: "t@t.com", role: "USER" })`
- Рендер `<Header />`
- Перевірка: `getByText("testuser")` знаходить елемент, `queryByText("Увійти")` → null

**Сценарій B — гість:**
- Передумова: `userStore.clear()`
- Рендер `<Header />`
- Перевірка: `getByText("Увійти")` знаходить елемент, `queryByText("testuser")` → null

#### I4: api.ts ↔ storage.ts

**Передумова:** `storage.setToken("test-jwt-token")`

**Кроки:**
